## 2. The composition recipe (instruments master → filter → batched `/quote`)

The only way to produce an option chain on Kite Connect:

```
Step 1 (daily, cold):  GET /instruments/NFO   (and /instruments/BFO for BSE F&O)
                       → gzipped CSV master dump
Step 2 (in-memory):    filter rows: name == <underlying>  AND  expiry == <chosen date>
                       AND instrument_type IN (CE, PE)   → the chain's legs
Step 3 (per refresh):  GET /quote?i=NFO:<sym1>&i=NFO:<sym2>&…   batched ≤500 legs/call
                       → per-leg LTP, OI, volume, 5-level depth, circuit limits
```

### Step 1 — the instruments master

- **Endpoints (Verified, LIVE-DOC runner 2026-07-14 + CLIENT-LIB-SOURCE `connect.py` `_routes`):** `GET /instruments` ("Retrieve the CSV dump of all tradable instruments") and `GET /instruments/:exchange` — first two rows of the live market-quotes endpoint table. SDK exchange constants include `NFO` (NSE F&O) and `BFO` (BSE F&O) (`connect.py` lines 80–86); BFO also appears as a live `exchanges` value in the user-profile example on https://kite.trade/docs/connect/v3/user/ (LIVE-DOC).
- **Format (Verified, LIVE-DOC + CLIENT-LIB-SOURCE + OFFICIAL-MOCK):** live-doc verbatim: *"Unlike the rest of the calls that return JSON, the instrument list API returns a gzipped CSV dump of instruments across all exchanges that can be imported into a database. The dump is generated once everyday and hence last_price is not real time."* (https://kite.trade/docs/connect/v3/market-quotes/). Header confirmed live AND by mock: `instrument_token, exchange_token, tradingsymbol, name, last_price, expiry, strike, tick_size, lot_size, instrument_type, segment, exchange`; `connect.py`'s response handler special-cases `"csv" in content-type` and `_parse_instruments` reads it with `csv.DictReader`.
- **Size (Verified, LIVE-DOC + CLIENT-LIB-SOURCE):** live page: *"Between multiple exchanges and segments, there are tens of thousands of different kinds of instruments"* + Warning: *"The instrument list API returns large amounts of data."* SDK docstring: *"several hundred KBs in size, with tens of thousands of entries"*.
- **Refresh discipline (Verified, LIVE-DOC runner 2026-07-14 — upgraded from SEARCH):** live Warning verbatim: *"It's best to request it once a day (ideally at around 08:30 AM) and store in a database at your end."* No numeric rate limit for `/instruments` is documented anywhere on the live page — the once-a-day Warning is the only stated policy (live-verified absence). **Compare: Dhan/Groww** — same daily-static-master pattern (Dhan Detailed CSV, `docs/dhan-ref/09-instrument-master.md`; Groww master CSV, `docs/groww-ref/09-instruments-csv.md`).
- **Identity wart (Verified, LIVE-DOC runner 2026-07-14 — upgraded from SEARCH; LOAD-BEARING):** live Note verbatim: *"For storage, it is recommended to use a combination of exchange and tradingsymbol as the unique key, not the numeric instrument token. Exchanges may reuse instrument tokens for different derivative instruments after each expiry."* The historical page independently repeats it: *"the exchanges flush the instrument_token for futures and options contracts for every expiry"* (https://kite.trade/docs/connect/v3/historical/). This is the Kite analog of the repo's I-P1-11 composite-key law and of Dhan's "derivative SecurityIds are unstable" rule (`.claude/rules/dhan/instrument-master.md` rule 3).

### Step 2 — filtering a chain out of the master

**Verified (LIVE-DOC + OFFICIAL-MOCK):** the live docs' own CSV sample carries F&O rows (note the EMPTY `name` column):

```csv

5720578,22346,NIFTY159500CE,,23.0,2015-12-31,9500,0.05,75,CE,NFO-OPT,NFO  (live docs sample)
12073986,47164,BANKNIFTY18JAN23500CE,,2528.4,2018-01-25,23500.0,0.05,40,CE,NFO-OPT,NFO   (mock)
```

- Chain selectors available per row: `expiry` (ISO date), `strike` (float), `instrument_type` (`CE`/`PE` — live column doc: "EQ, FUT, CE, PE"), `segment` (`NFO-OPT`), `lot_size`, `tick_size`. Expiry dates come from the master itself — enumerate `DISTINCT expiry` for the underlying; there is **no expiry-list endpoint** (Verified, LIVE-DOC: the five-endpoint table in §1 is the complete market-data surface; contrast Dhan's `/optionchain/expirylist`).
- **Underlying-matching wart (Verified, LIVE-DOC runner 2026-07-14 — upgraded from mock-only):** the live column table defines `name` as *"Name of the company (for equity instruments)"*, and the live docs' own NFO FUT/CE sample rows ship `name` EMPTY — so the underlying is recoverable from the `tradingsymbol` prefix (`NIFTY…`, `BANKNIFTY…`), matching the 2018 mock. Residual (minor): the docs sample is illustrative; measuring a full live NFO dump remains a day-0 nicety, but the docs-level definition already says name is equity-only. Robust filtering should treat tradingsymbol-prefix parsing as fallback, with the strike/expiry columns as the authoritative leg identity.
- **Assumed (order of magnitude):** an index chain is ~90–110 strikes × 2 legs ≈ **~200 instruments per expiry** — same magnitude the groww-ref pack assumes for the Groww chain (`docs/groww-ref/README.md` [R#15]). Not a Kite-verified number; actual strike counts come from the live master.

### Step 3 — batched `/quote` for the legs

- Full per-leg snapshot via `GET /quote` (see `07-market-quotes.md` §4): live-doc verbatim: *"the complete market data snapshot of up to 500 instruments in one go. It includes the quantity, OHLC, and Open Interest fields, and the complete bid/ask market depth"* — everything a chain view needs EXCEPT greeks/IV (§3 below).
- **Caps (Verified, LIVE-DOC runner 2026-07-14 — upgraded from SEARCH):** the live Limits table: `/quote` **500**, `/quote/ohlc` **1000**, `/quote/ltp` **1000** instruments per call (https://kite.trade/docs/connect/v3/market-quotes/). Rate: quote family = **1 request/second**, API-key-level (forum 13397 staff-thread rate table, live-fetched in the bundle; see `07-market-quotes.md` §3/§8).
- **ARITH (from the LIVE-DOC caps + forum rate):** one ~200-leg index chain (one expiry) = **1 `/quote` call** → refreshable at ~1 Hz best-case. Whole-chain-family math: N legs total across underlyings/expiries needs `⌈N/500⌉` calls at 1/s — e.g. 3 index underlyings × 1 expiry ≈ 600 legs ≈ 2 calls ≈ 2 s per full sweep; per-leg OHLC-only sweeps stretch to 1000/call. Derivation: cap × rate per `07-market-quotes.md` §3/§8.
- **Compare (cadence):** Dhan delivers the WHOLE chain (all strikes, greeks included) in ONE call per underlying at 1-per-3s (`docs/dhan-ref/06-option-chain.md`); Groww in ONE call per underlying+expiry inside the 10/s+300/min Live Data pool (`docs/groww-ref/14-option-chain.md`). Kite's composed chain costs more calls but returns per-leg DEPTH, which neither Dhan's nor Groww's chain payload carries (Groww chain has no per-leg bid/ask — [R#14]; Dhan chain carries top_bid/ask prices but not 5-level depth per `docs/dhan-ref/06-option-chain.md`).

### Streaming alternative for the composed chain

**Verified (LIVE-DOC runner 2026-07-14 + CLIENT-LIB-SOURCE):** the live WS page: *"You can subscribe for up to 3000 instruments on a single WebSocket connection … Single API key can have upto 3 websocket"* connections (https://kite.trade/docs/connect/v3/websocket/) — the 3000/conn cap is upgraded from SEARCH to LIVE-DOC. The `full` mode packet (184 bytes; depth at bytes 64–184 per the live page's market-depth structure) carries `oi`, `oi_day_high`, `oi_day_low`, `last_trade_time` + market depth (cross-check pykiteconnect `ticker.py` `_parse_binary`, lines 780–823) — so a composed chain's legs can be subscribed on the WS instead of polled over REST. Same composition step 1–2 applies; only step 3 changes transport.
