## Context / the gap (verified in CI history + reproduced locally)

Every `main` push since at least 2026-06-08 is red. Two independent chronic failures:

1. **`Coverage & Perf` (post-merge CI job)** fails with `error: 1 target failed: -p tickvault-api --lib` (exit 101) while the PR-time nextest `Test (api)` job passes the same commit. Root cause: 9 tests in `crates/api/src/handlers/debug.rs` mutate the process-global env vars `TICKVAULT_LOGS_DIR` / `TICKVAULT_SPILL_DIR` via `set_var`/`remove_var` with no serialization. nextest isolates one process per test (race invisible); the coverage job runs plain `cargo test` under cargo-llvm-cov — all tests are parallel threads of ONE process, so concurrent env mutation races. Reproduced locally: 3/60 loop runs FAILED pre-fix (`spill_status_reports_real_counts_when_dir_populated` panicked at debug.rs:478 reading `"exists":false` after a sibling swapped `SPILL_DIR_ENV`).
2. **`Claude Triage Loop`** red every 10 minutes: `claude-code-action@v1` removed the `direct_prompt`/`mcp_config`/`timeout_minutes` inputs (run logs print "Unexpected input(s)" and the prompt is silently ignored), and the `ANTHROPIC_API_KEY` repo secret is empty.

## Design

1. **Env-race fix (test-only, zero production code):** add `static ENV_LOCK: Mutex<()>` + poison-tolerant `env_guard()` helper to the `tests` module of `crates/api/src/handlers/debug.rs`; every one of the 9 env-mutating tests takes the guard as its first statement, serializing all access to the process-global environment regardless of test harness (cargo test threads OR nextest processes). The factually-wrong "tests run serially per #[test]" comment is corrected.
2. **Workflow input migration:** `claude-triage-loop.yml`, `claude-mobile-command.yml`, `claude-code-review.yml` migrate `direct_prompt:` → `prompt:`, `mcp_config: .mcp.json` → `claude_args: "--mcp-config .mcp.json"`, and drop `timeout_minutes:` in favour of job-level `timeout-minutes`.
3. **Fail-soft key guard (triage loop only):** a pre-step checks `ANTHROPIC_API_KEY` presence; when empty it emits a `::warning` annotation and skips the Claude step — converting 144 red scheduled runs/day into one loud actionable signal. The manual-dispatch workflows keep failing loudly (a human is watching those).

## Edge Cases

- **Poisoned lock:** a panicking env test would poison a plain `Mutex`; `env_guard()` uses `unwrap_or_else(PoisonError::into_inner)` so one failure cannot cascade-fail the other 8 tests.
- **Tokio tests holding a std guard across `.await`:** the 4 async env tests run single-task `current_thread` runtimes; the guard intentionally blocks the OS thread so concurrent env tests serialize — that is the fix, not a deadlock (no env test awaits another env test).
- **nextest path unchanged:** under process-per-test isolation the lock is uncontended; zero behavioural change for the green jobs.
- **Key present in future:** the guard step sets `present=true` and the Claude step runs exactly as before — no behaviour change once the operator adds the secret.
- **Scheduled event has no `head_commit`:** the prompt interpolation tolerates the empty timestamp (pre-existing behaviour, unchanged).

## Failure Modes

- If the race had a second source (not env vars), the post-merge `Coverage & Perf` run on main would still fail → caught by the PR-monitoring loop on #1072; the job re-runs plain `cargo test` under llvm-cov on every future main push, so any regression re-surfaces immediately.
- If `claude-code-action@v1` renames inputs again, the run logs print "Unexpected input(s)" warnings — same detection path used to diagnose this failure.
- A wrong fix cannot silently pass: pre-fix evidence (3/60 fail) vs post-fix (0/60 + 182/182 instrumented) brackets the change.
